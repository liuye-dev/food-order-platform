# 在线点餐平台开发约束

> 每次编写代码、修改配置、补充文档前，必须先阅读本文件，并确保改动没有偏离课程作业要求、需求分析报告和详细设计报告。

## 一、项目命名

- 项目中文名固定为**在线点餐平台**
- 项目代码名固定为 `food-order-platform`
- 页面标题、README、接口文档、编码实现报告中统一使用“在线点餐平台”

## 二、技术选型

| **层次** | **技术** | **约束** |
|---|---|---|
| 前端 | Vue、Vite | 实现顾客端和商家端核心页面 |
| 后端 | Django、Django REST Framework | 提供 REST API 和业务规则 |
| 数据库 | SQLite | 使用本地文件数据库，降低验收运行成本 |
| 依赖管理 | uv、pyproject.toml、uv.lock | Python 依赖以 `pyproject.toml` 和 `uv.lock` 为准 |
| 接口文档 | drf-spectacular | 用于接口演示和验收说明 |

## 三、MVP 主流程

必须优先跑通以下闭环：

```text
顾客登录
  -> 浏览菜单
  -> 选择商品规格
  -> 加入购物车
  -> 提交订单
  -> 模拟支付成功
  -> 商家查看订单
  -> 商家推进订单状态
  -> 顾客查看订单状态
```

| **优先级** | **功能** | **对应需求** | **实现要求** |
|---|---|---|---|
| P0 | 手机号快捷登录 | F01、US01 | 使用固定验证码或模拟验证码，新用户自动创建 |
| P0 | 菜单分类与搜索 | F02、US04 | 支持分类展示和商品名模糊搜索 |
| P0 | 商品规格选择 | F02、US05 | 支持规格选择并影响展示价格 |
| P0 | 购物车管理 | F03、US06 | 支持加入、数量修改、删除和总价计算 |
| P0 | 下单与模拟支付 | F03、US07 | 生成唯一订单号，支付成功后进入制作中 |
| P0 | 订单状态跟踪 | F04、US08 | 展示制作中、请取货、已完成等状态 |
| P0 | 商品上下架管理 | F05、US10 | 商家可编辑商品并设置售罄 |

| P2 | 积分与优惠券 | F01、US03 | 可使用静态展示或预留字段 |

## 五、范围控制

必须做：

- 顾客端点餐下单主流程
- 商家端商品管理和订单处理
- SQLite 数据持久化
- Django migration
- 基础演示数据
- 本地一键启动脚本

不做或只做模拟：

- 不接真实短信验证码
- 不接真实第三方支付
- 不做配送系统
- 不接门店 POS 硬件
- 不做复杂营销、推荐、满减规则
- 不为追求完整外卖系统而增加无关模块

## 六、数据库约束

- 数据库使用 SQLite
- 数据库连接参数可从 `.env` 或环境变量读取
- `.env` 不允许提交到 Git
- `.env.example` 必须提交，用于说明配置项
- 数据库表结构由 Django `models.py` 和 `migrations/` 维护
- 初始化演示数据使用 Django fixture
- 本地数据库文件 `backend/db.sqlite3` 不提交

配置示例：

```text
SQLITE_DB_PATH=backend/db.sqlite3
```

## 七、后端实现约束

- API 风格以 REST 为主
- 业务规则必须在后端校验，前端金额只用于展示
- 创建订单时必须重新校验商品是否售罄
- 订单明细必须保存商品名称、价格、规格快照
- 模拟支付成功后必须同步更新支付记录和订单状态
- 商家端只能按合法顺序推进订单状态
- 后端返回格式尽量保持统一

统一响应格式参考：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

## 八、前端实现约束

- 前端必须体现顾客端和商家端两个使用角色
- 先进入独立登录页，再根据登录角色进入对应页面
- 顾客端优先保证点餐链路完整
- 商家端优先保证商品售罄和订单状态推进
- 页面内容必须与“在线点餐平台”一致
- 前端状态变化必须能反映后端数据变化

## 九、完成代码后检查清单

- 后端接口能运行或至少通过语法检查
- 前端页面能启动或至少通过构建检查
- 数据库迁移文件与模型一致
- README 或报告需要同步时已经更新
- 功能能追溯到需求分析报告中的 Feature / User Story
