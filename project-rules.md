# 在线点餐平台开发约束

> 每次编写代码、修改配置、补充文档前，必须先阅读本文件，并确保改动没有偏离课程作业要求、需求分析报告和详细设计报告

## 一、项目命名

- 项目中文名固定为**在线点餐平台**
- 项目代码名固定为 `food-order-platform`
- 页面标题、README、接口文档、编码实现报告中统一使用“在线点餐平台”
- 不再使用旧项目名作为项目名或品牌名

## 二、依据文档

本项目实现必须追溯到以下 3 份文档：

| **文档** | **作用** |
|---|---|
| `HW2-项目作业要求.md` | 明确课程作业验收要求 |
| `HW2-需求分析报告.md` | 明确 Feature、User Story、用户角色和系统边界 |
| `HW2-详细设计报告.md` | 明确架构、数据模型、接口、活动图、类图和顺序图来源 |

## 三、硬性验收要求

- 必须使用 `Django + Vue`
- 必须具备数据持久化能力
- 数据库最终选型为 **MySQL**
- 必须支持关键业务流程操作
- 最终交付阶段使用 `Docker Compose` 打包前端、后端和 MySQL
- 编码实现报告必须说明项目简介、Prompt、Vibe Coding 过程、AI 生成后的修改、存在问题、功能与 User Story 的对应关系

## 四、技术选型

| **层次** | **技术** | **约束** |
|---|---|---|
| 前端 | Vue、Vite | 实现顾客端和商家端核心页面 |
| 后端 | Django、Django REST Framework | 提供 REST API 和业务规则 |
| 数据库 | MySQL | 本地开发可连接本机 MySQL，Docker 验收连接容器内 MySQL |
| 依赖管理 | uv、requirements.txt | Python 依赖以 `requirements.txt` 为准 |
| 接口文档 | drf-spectacular | 用于接口演示和验收说明 |
| 最终打包 | Docker、Docker Compose | 老师验收时不要求手动安装 Python、Node、MySQL |

## 五、MVP 主流程

必须优先跑通以下闭环：

```text
顾客登录
  -> 浏览菜单
  -> 选择商品规格
  -> 加入购物车
  -> 提交订单
  -> 模拟支付成功
  -> 商家查看订单
  -> 商家推进订单状态
  -> 顾客查看订单状态
```

## 七、范围控制

必须做：

- 顾客端点餐下单主流程
- 商家端商品管理和订单处理
- MySQL 数据持久化
- Django migration
- 基础演示数据
- Docker Compose 最终打包

不做或只做模拟：

- 不接真实短信验证码
- 不接真实第三方支付
- 不做配送系统
- 不接门店 POS 硬件
- 不做复杂营销、推荐、满减规则
- 不为追求完整外卖系统而增加无关模块

## 八、数据库约束

- 数据库使用 MySQL
- 本地开发阶段连接开发者本机 MySQL
- Docker 验收阶段连接 Docker Compose 内部 MySQL 服务
- 数据库连接参数必须从 `.env` 或环境变量读取
- `.env` 不允许提交到 Git
- `.env.example` 必须提交，用于说明配置项
- 数据库表结构由 Django `models.py` 和 `migrations/` 维护
- 初始化演示数据可使用 Django fixture 或 SQL 初始化脚本

本地开发数据库示例：

```text
DB_NAME=food_order_platform
DB_USER=root
DB_PASSWORD=本机 MySQL 密码
DB_HOST=127.0.0.1
DB_PORT=3306
```

Docker 验收数据库示例：

```text
DB_NAME=food_order_platform
DB_USER=food_order_user
DB_PASSWORD=food_order_pass
DB_HOST=mysql
DB_PORT=3306
```

## 九、Docker 约束

最终交付必须支持：

```powershell
docker compose up --build
```

Docker Compose 至少包含：

- `mysql`：MySQL 数据库服务
- `backend`：Django 后端服务
- `frontend`：Vue 前端服务

MySQL 初始化脚本建议放在：

```text
docker/mysql/init/
```

MySQL 数据持久化使用 Docker volume，不提交真实数据库数据目录

## 十、后端实现约束

- API 风格以 REST 为主
- 业务规则必须在后端校验，前端金额只用于展示
- 创建订单时必须重新校验商品是否售罄
- 订单明细必须保存商品名称、价格、规格快照
- 模拟支付成功后必须同步更新支付记录和订单状态
- 商家端只能按合法顺序推进订单状态
- 后端返回格式尽量保持统一

统一响应格式参考：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

## 十一、前端实现约束

- 前端必须体现顾客端和商家端两个使用角色
- 顾客端优先保证点餐链路完整
- 商家端优先保证商品售罄和订单状态推进
- 页面内容必须与“在线点餐平台”一致
- 不使用旧项目名作为页面标题或品牌名
- 前端状态变化必须能反映后端数据变化
